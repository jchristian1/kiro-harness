# task-model.md — Domain Model Contract

## Purpose

This document defines the complete domain model for kiro-worker: every entity, every field, every constraint, and every SQLite table definition. It is the contract that Phase 1 DB implementation must follow exactly.

**Cross-references:**
- Layer responsibilities → `architecture.md` § System Layers
- Task state names → `state-machine.md` § States (9 canonical states)
- API field usage → `worker-api.md` § Request/Response Schemas

## Migration Readiness

The schema is designed for SQLite in v1 and must migrate to Postgres without model changes. Rules that enforce this:

- All primary keys are UUIDs (TEXT in SQLite, UUID in Postgres — same values, no data migration)
- No SQLite-specific types are used (no BLOB for structured data; JSON is stored as TEXT)
- No SQLite-specific constraints that Postgres does not support
- `schema_version` is stored in a `meta` table; migrations are numbered and append-only
- Foreign key relationships are explicit and portable

```sql
CREATE TABLE meta (
  key   TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
INSERT INTO meta (key, value) VALUES ('schema_version', '1');
```

When migrating to Postgres: dump data, run equivalent `CREATE TABLE` statements with `UUID` and `JSONB` types substituted, reload data. No column renames or relationship changes required.

## Classification Enums

These enums are the canonical values for the Intent/Source/Operation model. All three dimensions are set at task creation and are immutable after that point.

### Intent

| Value | Meaning |
|---|---|
| `new_project` | Start a brand new project from scratch |
| `add_feature` | Add a feature to an existing project |
| `refactor` | Refactor or restructure existing code |
| `fix_bug` | Fix a specific bug or regression |
| `analyze_codebase` | Understand the repo without making changes |
| `upgrade_dependencies` | Update dependencies, handle breaking changes |
| `prepare_pr` | Prepare a pull request from existing work |

### Source

| Value | Meaning |
|---|---|
| `new_project` | No existing repo; worker creates workspace from scratch |
| `github_repo` | Clone a GitHub repository into a managed workspace |
| `local_repo` | Open an existing local git repository |
| `local_folder` | Open a local folder that may not be a git repo |

### Operation

| Value | Meaning | Approval required? |
|---|---|---|
| `plan_only` | Produce a plan, no code changes | No |
| `analyze_then_approve` | Analyze first, wait for approval before implementing | Yes (before implement) |
| `implement_now` | Implement immediately without analysis step | Only for destructive/push actions |
| `implement_and_prepare_pr` | Implement and prepare a PR | Yes (before push) |

### Task Status

The `status` field on a Task must be one of these 9 values. These are the canonical state names defined in `state-machine.md`. No other values are permitted.

| Value | Description |
|---|---|
| `created` | Task record exists; workspace not yet opened |
| `opening` | Worker is opening/cloning/validating the workspace |
| `analyzing` | Kiro CLI is running the analysis-workflow skill |
| `awaiting_approval` | Analysis complete; waiting for user approval to proceed |
| `implementing` | Kiro CLI is running the implementation-workflow skill |
| `validating` | Kiro CLI is running the validation-workflow skill |
| `awaiting_revision` | Validation found issues; waiting for user direction |
| `done` | Task complete; all runs stored; artifacts available |
| `failed` | Terminal failure; see `failure_reason` on the last Run |

### Run Mode

| Value | Description |
|---|---|
| `analyze` | Analysis invocation of Kiro CLI |
| `implement` | Implementation invocation of Kiro CLI |
| `validate` | Validation invocation of Kiro CLI |

### Run Status

| Value | Description |
|---|---|
| `running` | Kiro CLI subprocess is active |
| `completed` | Subprocess exited 0; output parsed and stored |
| `parse_failed` | Subprocess exited 0 but output was not valid JSON or failed schema validation |
| `error` | Subprocess exited non-zero or timed out |

### Artifact Type

| Value | Description |
|---|---|
| `analysis` | Output of an `analyze` run; conforms to analysis output schema |
| `implementation` | Output of an `implement` run; conforms to implementation output schema |
| `validation` | Output of a `validate` run; conforms to validation output schema |

## Entity: Project

A Project is the top-level registry entry. It maps a user-facing name to a workspace and tracks the active task.

### Field Schema

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID (TEXT) | Yes | Primary key. Generated by worker on creation. |
| `name` | TEXT | Yes | Human-readable project name. Unique per owner. |
| `source` | TEXT (Source enum) | Yes | Where the project code lives. Immutable after creation. |
| `source_url` | TEXT | No | GitHub URL or local path. Required for `github_repo`, `local_repo`, `local_folder`. Null for `new_project`. |
| `workspace_id` | UUID (TEXT) | No | FK → Workspace.id. The one active workspace for this project. Null until workspace is opened. |
| `owner_id` | TEXT | No | Reserved for multi-user support (Phase 11). Null in v1. All projects are implicitly owned by the single operator. |
| `created_at` | TEXT (ISO 8601) | Yes | UTC timestamp of project creation. |
| `updated_at` | TEXT (ISO 8601) | Yes | UTC timestamp of last modification. |

**owner_id note:** The field is present in the schema as a nullable column so that Phase 11 can populate it without a schema migration. In v1, it is always NULL. When multi-user support is added, a migration will backfill `owner_id` from a new `users` table and add a NOT NULL constraint.

### JSON Example

```json
{
  "id": "proj_01HZ3K8VXQM2N4P7R9T6W0YBCD",
  "name": "storefront-api",
  "source": "github_repo",
  "source_url": "https://github.com/acme/storefront-api",
  "workspace_id": "ws_01HZ3K9FXQM2N4P7R9T6W0YBCE",
  "owner_id": null,
  "created_at": "2025-07-14T09:00:00Z",
  "updated_at": "2025-07-14T11:32:00Z"
}
```

### SQLite Table Definition

```sql
CREATE TABLE projects (
  id           TEXT PRIMARY KEY,
  name         TEXT NOT NULL,
  source       TEXT NOT NULL CHECK (source IN ('new_project', 'github_repo', 'local_repo', 'local_folder')),
  source_url   TEXT,
  workspace_id TEXT REFERENCES workspaces(id) ON DELETE SET NULL,
  owner_id     TEXT,
  created_at   TEXT NOT NULL,
  updated_at   TEXT NOT NULL,
  UNIQUE (name)
);
```

## Entity: Workspace

A Workspace is the filesystem location where a project's code lives. In v1, there is exactly one active workspace per project. The one-active-workspace constraint is enforced at the application layer (see below) and by the FK from `projects.workspace_id`.

### One-Active-Workspace-Per-Project Constraint

**Rule:** At any point in time, `projects.workspace_id` points to at most one Workspace record. The worker enforces this by:

1. When opening a workspace for a project that already has `workspace_id` set, the worker returns the existing workspace rather than creating a new one.
2. When a workspace is archived or deleted, `projects.workspace_id` is set to NULL before the workspace record is removed.
3. There is no `workspaces.is_active` flag — the single-workspace invariant is maintained by the FK pointer on the project, not by a flag on the workspace.

Multi-workspace per project is not modeled in v1. If a future phase requires it (e.g., feature branches), the model will add a `workspaces.project_id` FK and a separate `active_workspace_id` on projects, with a migration.

### Field Schema

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID (TEXT) | Yes | Primary key. |
| `project_id` | UUID (TEXT) | Yes | FK → Project.id. The project this workspace belongs to. |
| `path` | TEXT | Yes | Absolute filesystem path to the workspace root. Must be within the configured safe root. |
| `git_remote` | TEXT | No | Git remote URL if the workspace is a git repo. Null for `new_project` and `local_folder` sources. |
| `git_branch` | TEXT | No | Current git branch. Null for non-git workspaces. |
| `created_at` | TEXT (ISO 8601) | Yes | UTC timestamp of workspace creation. |
| `last_accessed_at` | TEXT (ISO 8601) | Yes | UTC timestamp of last worker access (open, invocation, etc.). |

### JSON Example

```json
{
  "id": "ws_01HZ3K9FXQM2N4P7R9T6W0YBCE",
  "project_id": "proj_01HZ3K8VXQM2N4P7R9T6W0YBCD",
  "path": "/var/kiro-worker/workspaces/storefront-api",
  "git_remote": "https://github.com/acme/storefront-api",
  "git_branch": "main",
  "created_at": "2025-07-14T09:01:00Z",
  "last_accessed_at": "2025-07-14T11:30:00Z"
}
```

### SQLite Table Definition

```sql
CREATE TABLE workspaces (
  id               TEXT PRIMARY KEY,
  project_id       TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  path             TEXT NOT NULL UNIQUE,
  git_remote       TEXT,
  git_branch       TEXT,
  created_at       TEXT NOT NULL,
  last_accessed_at TEXT NOT NULL
);
```

## Entity: Task

A Task represents a single unit of work requested by the user. It carries the classification (Intent/Source/Operation), the current lifecycle state, and links to all runs produced during its execution.

### Field Schema

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID (TEXT) | Yes | Primary key. |
| `project_id` | UUID (TEXT) | Yes | FK → Project.id. |
| `workspace_id` | UUID (TEXT) | Yes | FK → Workspace.id. Snapshot of the workspace at task creation. |
| `intent` | TEXT (Intent enum) | Yes | What the user wants to accomplish. Immutable after creation. |
| `source` | TEXT (Source enum) | Yes | Where the code lives. Immutable after creation. |
| `operation` | TEXT (Operation enum) | Yes | How the work should be executed. Immutable after creation. |
| `description` | TEXT | Yes | Free-text description of the task as provided by the user. |
| `status` | TEXT (Task Status enum) | Yes | Current lifecycle state. Must be one of the 9 canonical states. |
| `approved_at` | TEXT (ISO 8601) | No | UTC timestamp of when the task was approved. Null until `POST /tasks/{id}/approve` is called. |
| `created_at` | TEXT (ISO 8601) | Yes | UTC timestamp of task creation. |
| `updated_at` | TEXT (ISO 8601) | Yes | UTC timestamp of last status change. |

### JSON Example

```json
{
  "id": "task_01HZ3KA2XQM2N4P7R9T6W0YBCF",
  "project_id": "proj_01HZ3K8VXQM2N4P7R9T6W0YBCD",
  "workspace_id": "ws_01HZ3K9FXQM2N4P7R9T6W0YBCE",
  "intent": "add_feature",
  "source": "github_repo",
  "operation": "analyze_then_approve",
  "description": "Add JWT-based authentication to the storefront API. Users should be able to register, log in, and receive a token. Protected routes should reject requests without a valid token.",
  "status": "awaiting_approval",
  "approved_at": null,
  "created_at": "2025-07-14T09:05:00Z",
  "updated_at": "2025-07-14T09:47:00Z"
}
```

### SQLite Table Definition

```sql
CREATE TABLE tasks (
  id           TEXT PRIMARY KEY,
  project_id   TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  workspace_id TEXT NOT NULL REFERENCES workspaces(id),
  intent       TEXT NOT NULL CHECK (intent IN (
                 'new_project', 'add_feature', 'refactor', 'fix_bug',
                 'analyze_codebase', 'upgrade_dependencies', 'prepare_pr'
               )),
  source       TEXT NOT NULL CHECK (source IN (
                 'new_project', 'github_repo', 'local_repo', 'local_folder'
               )),
  operation    TEXT NOT NULL CHECK (operation IN (
                 'plan_only', 'analyze_then_approve', 'implement_now', 'implement_and_prepare_pr'
               )),
  description  TEXT NOT NULL,
  status       TEXT NOT NULL CHECK (status IN (
                 'created', 'opening', 'analyzing', 'awaiting_approval',
                 'implementing', 'validating', 'awaiting_revision', 'done', 'failed'
               )),
  approved_at  TEXT,
  created_at   TEXT NOT NULL,
  updated_at   TEXT NOT NULL
);

CREATE INDEX idx_tasks_project_id ON tasks(project_id);
CREATE INDEX idx_tasks_status ON tasks(status);
```

## Entity: Run

A Run is a single Kiro CLI invocation. Each task may have multiple runs (one per phase: analyze, implement, validate; plus retries). Runs are append-only — they are never updated after creation, only the `status` and output fields are filled in when the subprocess completes.

### Field Schema

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID (TEXT) | Yes | Primary key. |
| `task_id` | UUID (TEXT) | Yes | FK → Task.id. |
| `mode` | TEXT (Run Mode enum) | Yes | Which phase this run represents: `analyze`, `implement`, or `validate`. |
| `status` | TEXT (Run Status enum) | Yes | Current status of this run. |
| `agent` | TEXT | Yes | Name of the Kiro custom agent used (e.g., `repo-engineer`). |
| `skill` | TEXT | Yes | Name of the Kiro skill invoked (e.g., `analysis-workflow`). |
| `context_snapshot` | TEXT (JSON) | Yes | The full context JSON passed to Kiro CLI at invocation time. Stored for audit and resume. |
| `raw_output` | TEXT | No | Raw stdout from the Kiro CLI subprocess. Null until run completes. |
| `parse_status` | TEXT | No | `ok`, `parse_failed`, or `schema_invalid`. Null until run completes. |
| `failure_reason` | TEXT | No | Human-readable failure description. Null unless `status` is `error` or `parse_status` is not `ok`. |
| `progress_message` | TEXT | No | Latest human-readable progress message extracted from streaming stdout. Updated during execution. Null until first progress line is received. |
| `last_activity_at` | TEXT (ISO 8601) | No | UTC timestamp of the last progress update. Updated whenever `progress_message` is written. Null until first progress line. |
| `partial_output` | TEXT | No | Last ~2000 chars of stdout captured so far. Updated during streaming execution. Null until output starts arriving. |
| `started_at` | TEXT (ISO 8601) | Yes | UTC timestamp when the subprocess was launched. |
| `completed_at` | TEXT (ISO 8601) | No | UTC timestamp when the subprocess exited. Null while running. |

### JSON Example

```json
{
  "id": "run_01HZ3KB5XQM2N4P7R9T6W0YBCG",
  "task_id": "task_01HZ3KA2XQM2N4P7R9T6W0YBCF",
  "mode": "analyze",
  "status": "completed",
  "agent": "repo-engineer",
  "skill": "analysis-workflow",
  "context_snapshot": {
    "task_id": "task_01HZ3KA2XQM2N4P7R9T6W0YBCF",
    "intent": "add_feature",
    "source": "github_repo",
    "operation": "analyze_then_approve",
    "description": "Add JWT-based authentication to the storefront API.",
    "workspace_path": "/var/kiro-worker/workspaces/storefront-api",
    "current_status": "analyzing",
    "prior_analysis": null,
    "approved_plan": null,
    "revision_instructions": null
  },
  "raw_output": "{\"schema_version\":\"1\",\"mode\":\"analyze\",\"headline\":\"JWT auth is feasible with moderate effort\", ...}",
  "parse_status": "ok",
  "failure_reason": null,
  "started_at": "2025-07-14T09:10:00Z",
  "completed_at": "2025-07-14T09:47:00Z"
}
```

### SQLite Table Definition

```sql
CREATE TABLE runs (
  id                TEXT PRIMARY KEY,
  task_id           TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  mode              TEXT NOT NULL CHECK (mode IN ('analyze', 'implement', 'validate')),
  status            TEXT NOT NULL CHECK (status IN ('running', 'completed', 'parse_failed', 'error')),
  agent             TEXT NOT NULL,
  skill             TEXT NOT NULL,
  context_snapshot  TEXT NOT NULL,
  raw_output        TEXT,
  parse_status      TEXT CHECK (parse_status IN ('ok', 'parse_failed', 'schema_invalid')),
  failure_reason    TEXT,
  progress_message  TEXT,
  last_activity_at  TEXT,
  partial_output    TEXT,
  started_at        TEXT NOT NULL,
  completed_at      TEXT
);

CREATE INDEX idx_runs_task_id ON runs(task_id);
```

**Progress fields note:** `progress_message`, `last_activity_at`, and `partial_output` are written during streaming kiro-cli execution via the `on_progress` callback. They allow `GET /tasks/{id}` to return live progress while a run is in flight. WAL mode (`PRAGMA journal_mode = WAL`) is required so these writes are visible to concurrent readers without blocking the writer.

## Entity: Artifact

An Artifact is the parsed, validated output of a completed Run. It stores the structured JSON that conforms to one of the three output schemas defined in `kiro-output-contract.md`. Artifacts are the source of truth for resume context reconstruction.

### Field Schema

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID (TEXT) | Yes | Primary key. |
| `run_id` | UUID (TEXT) | Yes | FK → Run.id. One artifact per completed run. |
| `task_id` | UUID (TEXT) | Yes | FK → Task.id. Denormalized for fast lookup without joining through runs. |
| `type` | TEXT (Artifact Type enum) | Yes | Which output schema this artifact conforms to. |
| `schema_version` | TEXT | Yes | The `schema_version` value from the artifact JSON. Used for migration compatibility checks. |
| `content` | TEXT (JSON) | Yes | Full parsed artifact JSON. Conforms to the schema identified by `type` and `schema_version`. |
| `file_path` | TEXT | No | Optional filesystem path if the artifact is also written to disk (for large outputs). Null if content is stored inline only. |
| `created_at` | TEXT (ISO 8601) | Yes | UTC timestamp of artifact creation. |

### JSON Example

```json
{
  "id": "art_01HZ3KC8XQM2N4P7R9T6W0YBCH",
  "run_id": "run_01HZ3KB5XQM2N4P7R9T6W0YBCG",
  "task_id": "task_01HZ3KA2XQM2N4P7R9T6W0YBCF",
  "type": "analysis",
  "schema_version": "1",
  "content": {
    "schema_version": "1",
    "mode": "analyze",
    "headline": "JWT auth is feasible with moderate effort; 4 files need changes",
    "findings": [
      "No existing auth middleware found",
      "Express 4.18 is in use; jsonwebtoken 9.x is compatible",
      "User model exists in src/models/user.js but has no password field"
    ],
    "affected_areas": ["src/models/user.js", "src/routes/auth.js (new)", "src/middleware/auth.js (new)", "src/app.js"],
    "tradeoffs": ["Adding bcrypt increases install size by ~1MB", "Stateless JWT means no server-side session revocation without a blocklist"],
    "risks": ["Password hashing must use bcrypt with cost factor ≥ 12", "Token expiry must be set; default to 24h"],
    "implementation_steps": [
      "Add password field to User model with bcrypt hashing hook",
      "Create POST /auth/register and POST /auth/login routes",
      "Create JWT middleware that validates Bearer tokens",
      "Apply middleware to all non-public routes"
    ],
    "validation_commands": ["npm test", "npm run lint"],
    "questions": [],
    "recommended_next_step": "approve_and_implement"
  },
  "file_path": null,
  "created_at": "2025-07-14T09:47:00Z"
}
```

### SQLite Table Definition

```sql
CREATE TABLE artifacts (
  id             TEXT PRIMARY KEY,
  run_id         TEXT NOT NULL REFERENCES runs(id) ON DELETE CASCADE,
  task_id        TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  type           TEXT NOT NULL CHECK (type IN ('analysis', 'implementation', 'validation')),
  schema_version TEXT NOT NULL,
  content        TEXT NOT NULL,
  file_path      TEXT,
  created_at     TEXT NOT NULL,
  UNIQUE (run_id)
);

CREATE INDEX idx_artifacts_task_id ON artifacts(task_id);
```

## Entity Relationships

```
Project (1) ──── (0..1) Workspace
Project (1) ──── (0..N) Task
Task    (1) ──── (0..N) Run
Run     (1) ──── (0..1) Artifact
```

- A Project has at most one active Workspace in v1 (enforced via `projects.workspace_id` FK).
- A Project may have many Tasks over its lifetime, but only one Task should be in a non-terminal state at a time (application-layer convention; not a DB constraint in v1).
- A Task has one Run per phase (analyze, implement, validate), plus additional Runs for retries. Runs are ordered by `started_at`.
- A Run has at most one Artifact (created only when `parse_status = 'ok'`).

## Resume Context Object

When the worker resumes a task, it constructs this context object from its DB and passes it to Kiro CLI as the `--context` parameter. This is the complete set of fields required for a stateless Kiro invocation to continue work correctly.

```json
{
  "task_id": "task_01HZ3KA2XQM2N4P7R9T6W0YBCF",
  "intent": "add_feature",
  "source": "github_repo",
  "operation": "analyze_then_approve",
  "description": "Add JWT-based authentication to the storefront API.",
  "workspace_path": "/var/kiro-worker/workspaces/storefront-api",
  "current_status": "awaiting_approval",
  "prior_analysis": {
    "schema_version": "1",
    "mode": "analyze",
    "headline": "JWT auth is feasible with moderate effort; 4 files need changes",
    "implementation_steps": ["..."],
    "recommended_next_step": "approve_and_implement"
  },
  "approved_plan": null,
  "revision_instructions": null
}
```

**Rule:** The worker constructs this object exclusively from its own DB (task record + artifact content). It does not rely on Kiro session history. Every Kiro invocation is stateless from Kiro's perspective.

**SQLite connection pragmas:** Phase 1 implementation must execute the following on every SQLite connection before any query:
```sql
PRAGMA foreign_keys = ON;
PRAGMA journal_mode = WAL;
```
`foreign_keys = ON` enforces referential integrity (SQLite does not enforce FKs by default). `journal_mode = WAL` enables concurrent readers and writers — essential for progress updates written during a long run to be visible to concurrent `GET /tasks/{id}` requests.
